明白了，您希望严格区分“人可读的逻辑确认”与“机可执行的技术规格”。
clarify 命令的职责就是充当“翻译记录官”：它只负责把老板/客户口中的“大白话”，整理成一份结构清晰、无歧义的 Markdown 文档。这份文档是业务人员和技术人员都能看懂的“中间态协议”。
只有当人类在这个 Markdown 上点了头（确认无误），下一个命令 formalize 才会拿着这份 Markdown 去生成 SQL、DDL 和血缘图。
基于这个要求，重新设计如下：
🛠️ 命令名称：clarify
核心定位：将模糊的对话意图，转化为一份结构化的 Markdown 需求澄清文档。它是业务逻辑的“最终裁决版”，但不包含任何技术实现细节。
1. 命令语法
clarify <raw_intent> -o <output_file.md>
参数说明：
raw_intent: 原始的模糊需求（字符串或引用）。
-o, —output: 必填。指定输出的 Markdown 文件路径。
—interactive: 是否开启交互模式（默认开启）。如果关闭，AI 将基于常见惯例直接生成文档，人工需事后审核。
2. 执行流程
意图分析：提取关键词（如“销售额”、“增长”）。
生成澄清草案：AI 在内部模拟专家提问，基于业务知识库生成假设的逻辑描述。
文档生成：将分析结果填充进标准的 Markdown 模板。
产出交付：保存为 .md 文件，流程结束。（注意：这里不涉及任何数据库 Schema 查询，不生成 JSON Schema，也不生成 SQL）
3. 产出物模板：需求澄清文档
这是 clarify 命令生成的 .md 文件内容。它是一份“业务逻辑说明书”。
# 需求澄清单：销售增长分析
> **创建时间**: 2023-10-27 10:00
> **原始意图**: ”我要看上个季度的销售增长趋势，按大区拆分。“
> **状态**: 待确认
—
## 1. 核心指标定义
| 指标名称 | 业务语义定义 | 计算逻辑描述 | 备注 |
| :— | :— | :— | :— |
| **销售额** | **实收金额** | 客户实际支付的订单金额，扣除退款、扣减项。 | 不包含未发货订单金额。 |
| **增长趋势** | **环比增长率** | (本期销售额 - 上期销售额) / 上期销售额 × 100% | 对比周期为：本月 vs 上月。 |

## 2. 分析维度
*   **时间维度**: **季度**
    *   范围：2023年 Q3 (7月-9月)。
*   **地理维度**: **销售大区**
    *   粒度：一级大区（如：华东大区、华北大区），不拆解到省份。

## 3. 数据范围与过滤
*   **业务线**: 全产品线（默认）。
*   **渠道**: 线上+线下全渠道。
*   **排除项**: 排除测试订单、取消订单。

## 4. 澄清历史记录
*   **Q**: 销售额是指 GMV 还是实收？
    *   **A**: 实收。
*   **Q**: 增长是同比还是环比？
    *   **A**: 环比。
4. 演示案例
场景：
产品经理在终端输入需求。
执行命令：
# 系统生成一份 Markdown 文件，暂存到本地或 Git 仓库
clarify ”我要看最近一个月APP的活跃用户，去重算，按城市排名前10“ -o ./reqs/mau_ranking_v1.md
系统行为（AI 幕后思考）：
思考：“活跃用户”通常指 DAU 或 MAU，语境是“最近一个月”，倾向于 MAU。
思考：“去重算”意味着 Count(Distinct)。
思考：“城市排名”意味着需要按城市分组并 Order By。
产出文件内容 (mau_ranking_v1.md):
# 需求澄清单：城市活跃度排名
> **原始意图**: ”我要看最近一个月APP的活跃用户，去重算，按城市排名前10“
## 1. 指标定义
*   **活跃用户数**: 最近 30 天内，启动过 APP 的去重用户数。
## 2. 维度与排序
*   **城市**: 用户注册地或常驻地（以最后一次登录为准）。
*   **排序**: 按活跃用户数**降序**排列。
*   **截取**: 仅展示排名 **Top 10** 的城市。
## 3. 时间范围
*   **统计周期**: 自然月还是过去30天？
    *   *澄清结果*: 采用 **过去 30 天 (Rolling 30 Days)**。
5. 关键点解析：为什么是 Markdown？
人类友好：
Markdown 是纯文本，GitHub/GitLab 原生支持，对比差异方便。
业务人员看 JSON 会头晕，但看 Markdown 表格毫无压力。
解耦业务与技术：
clarify 产出的文档里，绝对没有 user_id (字段名)、ods_user_log (表名)、BigQuery (数据库类型)。
它只关心“业务逻辑”（什么是活跃，怎么算排名）。
为 formalize 做准备：
当业务主管在这个 Markdown 文件上评论一句：“确认，没问题。”
formalize 命令就可以读取这个文件，利用 LLM 理解其中的“活跃用户”、“过去30天”，然后去技术库里查找对应的字段并生成代码。
这样，clarify 是“定规矩”（业务侧），formalize 是“做工程”（技术侧），边界切割得非常干净。
